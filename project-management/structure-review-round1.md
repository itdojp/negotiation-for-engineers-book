# 1. 書籍の概要

- 想定読者（本文から読み取れる範囲）
  - 技術的な提案（技術選定、見積もり、改善投資、プロセス改善など）が「技術的に正しい」だけでは通らず、意思決定者・非技術職・他チームとの合意形成に課題を感じているエンジニア。
  - 経営層／プロダクトマネージャー／他エンジニア／非技術職など、ステークホルダーごとに説明の仕方を変える必要がある立場のエンジニア。
- 本書で達成しようとしているゴール（`docs/index.md` の記述ベース）
  - エンジニアの論理的思考・エンジニアリング能力を、ビジネス交渉の場で再現性のある手法として活用できるようにする。
  - 交渉を「一回限りの説得」ではなく、「準備→実行→フォローアップ」の反復可能なプロセス（パイプライン）として設計し、継続的改善できる状態にする。

## 2. 現状の章構成サマリ

## はじめに：なぜエンジニアこそ交渉力が必要か

- 章の狙い / 主なトピック
  - エンジニアが直面する交渉シーン（技術選定、見積もり、予算獲得、チーム内合意）を提示し、技術力と交渉力の相乗効果を説明する。
  - 本書全体のロードマップ（各章の役割）と読み方の指針を提示する。
- 章頭の学習目標（要約）
  - 明示的な「学習目標」見出しはないが、交渉課題の典型パターンと、本書の学習ルートを把握することが主目的。

## 第1章：コードは雄弁に語る - 技術的根拠による説得術

- 章の狙い / 主なトピック
  - 技術的な正しさを「相手が納得できる根拠」に変換するための武器として、プロトタイプ・データ・オープンソース活用を扱う。
  - コード／デモ／メトリクスといった具体物で、合意形成の摩擦を下げるアプローチを示す。
- 章頭の学習目標（要約）
  - 学習目標の箇条書きはないが、導入文から「3つの武器を使い分け、説得材料に落とす」ことが狙い。

## 第2章：ステークホルダー・インターフェース設計

- 章の狙い / 主なトピック
  - 経営層／プロダクトチーム／他エンジニア／非技術職それぞれに対して、説明粒度・判断軸・コミュニケーション方法を最適化する。
  - 「技術→ビジネス言語への変換」「非同期コミュニケーション」「技術討論の構造化」などを扱う。
- 章頭の学習目標（要約）
  - 明示的な学習目標はないが、「相手ごとのプロトコル設計（変換・伝達・合意）」が中核。

## 第3章：アジャイル交渉術 - 反復的合意形成

- 章の狙い / 主なトピック

- 章頭の学習目標（要約）

## 第6章：交渉パイプラインの構築

## 3. 構成上の気になる点（候補）

- 章の学習目標の明示度
  - 各章の導入・章末のまとめはある一方で、章頭に「この章でできるようになること」が明示されていないため、Stage2要件（学習目標と節内容の対応）の自己チェックがしづらい。
- 「導入 → 本論 → まとめ」の視認性
  - 章末のまとめは段落で書かれているが、見出しとして明示されていないため、読者が「どこを読み終えたらよいか」を把握しにくい可能性がある（特にボリュームの大きい章）。
- 付録／テンプレートの役割分担
  - 第6章に「テンプレート集」があり、付録にもテンプレート・ログ・チェックリストがあるため、読者が「最終的にどれを使えばよいか」を迷う可能性がある（重複というより、配置と導線の課題）。
- 章間の参照導線（どこから何を使うか）
  - 章末まとめから付録（ツールキット）への参照が明示されていないため、「読む→使う」導線が弱くなる可能性がある。
- リポジトリ内の本文ソースが二重化している可能性
  - リポジトリ直下に `negotiation-chapter-*.md` 等があり、`src/**` にも同等の章がある。どちらが正（編集対象）かが不明確だと、将来的な差分混入や更新漏れのリスクになる（要確認）。

## 4. 構成改善の提案（案）

- 章頭に「学習目標（できるようになること）」を追加する
  - 各章冒頭に 3〜5 点程度で、章の到達目標を明示する（本文の導入段落を要約して箇条書き化するだけでも効果が高い）。
  - 可能なら、節見出し（例: 1.1〜1.3）と学習目標の対応が分かる「章内マップ（表）」を追加する。
- 章末まとめを「見出し＋箇条書き」で固定化する
  - 章末に `## まとめ`（または同等）を置き、学習目標の再掲（チェックリスト形式）と「次に読む章／付録」を案内する。
- 付録（ツールキット）と第6章のテンプレートの役割を明確化する
  - 【案】第6章は「テンプレートの使いどころ」と最小例に絞り、完全版テンプレート・チェックリストは付録に集約する。
  - 【案】付録側に「どの章を読んだら、どのテンプレートを使うか」対応表を置く。
- 章間の導線を強化する
  - 各章末のまとめに「推奨の次アクション（例: 付録のテンプレートを使って自分の案件で書いてみる）」を追記し、実務適用の手順を明確化する。
- 本文ソースの正本を明確化する（運用面）
  - `src/**` を正本とする（または直下の `negotiation-*.md` を正本とする）方針を明記し、片方は移動（例: `manuscript/legacy/`）または削除する案を検討する（要確認）。
